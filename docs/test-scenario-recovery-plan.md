# 테스트 시나리오 생성 복구 계획

## 현재 증상

테스트 시나리오 생성이 0%에서 멈추며 다음 오류를 반환한다.

```text
sourcePath, artifactRoot, streamId, and catalogFlows are required
```

## 확인된 원인

`POST /api/wiki/test-scenarios`는 네 입력을 하나의 조건으로 검사하고, 그중 하나라도 없으면 같은 오류를 반환한다. 공개 저장소의 기본 `flows/catalog.yaml`은 의도적으로 `flows: []`이므로 정상적인 오픈소스 설치에서도 `catalogFlows`가 빈 배열이 된다. 반면 현재 라우트는 빈 카탈로그를 오류로 처리한다.

즉, 조직별 선택 설정인 flow catalog가 테스트 시나리오 생성의 필수 계약으로 잘못 승격되어 있다. UI에서는 프로젝트 경로와 산출물 경로를 확보했더라도 빈 catalog 때문에 작업이 시작되지 않으며, 통합 오류 메시지 때문에 어떤 입력이 문제인지도 알 수 없다.

## 수정 범위

### 1. 입력 계약 분리

- `sourcePath`, `artifactRoot`, `streamId`를 각각 검증하고 필드별 오류 코드와 사용자 메시지를 반환한다.
- `catalogFlows`는 선택 입력으로 변경한다.
- 원본 저장소와 위키 산출물 경로가 같은 경우는 현재와 같이 차단한다.

### 2. 위키 산출물 기반 fallback

- catalog가 비어 있으면 `artifactRoot/business-flows/*.md`를 탐색한다.
- 파일명과 문서 제목에서 flow ID와 이름을 추출해 최소 `CatalogFlow`를 만든다.
- 문서에서 확인할 수 없는 repo, table, code reference는 빈 배열로 두고 프롬프트에서 “근거 없음”으로 취급한다.
- 문서가 하나도 없을 때만 “먼저 비즈니스 흐름 위키를 생성해 주세요”라는 실행 가능한 오류를 제공한다.

### 3. UI 사전 점검

- 실행 버튼을 누르기 전에 프로젝트 경로, 산출물 경로, business-flow 문서 수를 표시한다.
- catalog가 없으면 오류 대신 “생성된 위키 문서에서 흐름을 추론합니다” 상태를 보여준다.
- 백엔드 작업을 만들기 전 실패는 0% 진행률로 남기지 않고 `준비 실패` 상태로 종료한다.

### 4. 신뢰성 및 회귀 테스트

- 기본 공개 설정처럼 catalog가 비어 있는 경우
- catalog 없이 business-flow 문서만 있는 경우
- catalog와 문서가 일부만 일치하는 경우
- source/artifact 경로가 누락되거나 상대 경로인 경우
- 기존 캐시에서 열린 프로젝트인 경우
- 백엔드 작업 생성이 실패했을 때 재시도 가능한 경우

## 완료 기준

- 새로 설치한 공개 버전의 빈 `flows/catalog.yaml` 상태에서도 생성된 위키 문서만으로 테스트 시나리오 작업이 시작된다.
- 누락 입력은 필드별로 안내되며 통합 오류 문구가 더 이상 노출되지 않는다.
- 사전 검증 실패는 진행 중으로 남지 않고 사용자가 수정 후 다시 실행할 수 있다.
- catalog가 설정된 기존 사용자 흐름과 교차-flow 분석은 회귀하지 않는다.

## 우선순위

RepoLume 리브랜딩과 설치 패키지 검증을 먼저 완료한다. 이 문서의 변경은 후속 작업으로 구현하며, 현재 릴리스에서는 테스트 시나리오 기능의 알려진 제한으로 명시한다.
